Integrity**. If an attacker has compromised userspace (e.g., via a library pre-load rootkit), they can easily hide their presence from standard tools like `ps` or `netstat`. Zenith-Sentry solves this by moving its \"eyes\" into the kernel.\n\n### Tracepoints vs. Kprobes\n\nZenith-Sentry utilizes a hybrid approach to kernel monitoring:\n\n1.  **Tracepoints (`sys_enter_execve`)**: These are static hooks established in the Linux kernel source. Unlike dynamic kprobes, tracepoints are stable across kernel versions and provide a predictable API. We use these for process execution tracking because they capture the exact arguments being passed to the `execve` syscall before the process even begins execution. This is critical for preventing \"Time of Check/Time of Use\" (TOCTOU) exploits where an attacker changes the binary on disk after the monitor has checked it.\n2.  **Kprobes (`tcp_v4_connect`)**: Networking events are more dynamic. We use kprobes to hook into the TCP stack's connection initiation logic. This allows us to see outbound connection attempts the moment they are triggered, providing a \"wire-speed\" view of Command & Control (C2) activity.\n\n### Lock-Free Multi-Producer Ring Buffers\n\nTo ensure minimal impact on system performance, Zenith-Sentry uses **Lock-Free Ring Buffers** (specifically `BPF_PERF_OUTPUT` and `BPF_RINGBUF_OUTPUT`). \n\nIn high-concurrency environments, traditional mutex locks cause \"context switch storms\" that can degrade system performance. By using lock-free rings, the kernel can \"fire and forget\" security events into a shared memory space that the Zenith-Sentry userspace engine polls. This results in an overhead of less than 2 microseconds per event, making Zenith-Sentry suitable for high-transaction servers.\n\n### Metadata Structs & Event Packaging\n\nIn `zenith/ebpf/execve_monitor.c`, we define precise C structs for event packaging. This ensures binary compatibility between the kernel and userspace:\n\n```c\nstruct execve_event_t {\n    u32 pid;          // Process ID\n    u32 tgid;         // Thread Group ID (User-perceived PID)\n    u32 uid;          // User ID (Critical for root detection)\n    char comm[16];    // Executable Name\n    char filename[256]; // Full path of target binary\n};\n```\n\nThis structure ensures that every piece of data required for a forensic audit is captured in a single atomic kernel operation.\n\n---\n\n## Technical Thesis 2: Dynamic Plugin Orchestration\n\n```\n The engine itself should not need to know *how* to detect a threat; it should only know how to provide the *telemetry* to specialized plugins.\n\n### The Plugin Registry Factory\n\nIn `zenith/registry.py`, we employ Python's introspection capabilities (`importlib` and `inspect`) to build a dynamic plugin loader. \n\n1.  The registry scans the `zenith/plugins/` directory.\n2.  It dynamically imports every `.py` file found within.\n3.  It searches the module for any class that inherits from the `IDetector` base class (defined in `zenith/core.py`).\n4.  It constructs an instance of each found class, injecting the required telemetry data.\n\n### Introspective Telemetry Injection\n\nWhen the `ZenithEngine` (in `zenith/engine.py`) prepares to run a scan, it executes a \"Harvest\" phase. It gathers data from all collectors (Process, Network, System, eBPF) and then **Injects** this data into the plugins.\n\nThis dependency-injection model allows plugin developers to focus purely on the detection logic without worrying about how to extract data from the system.\n\n### Validation & Error Isolation\n\nTo prevent a single buggy plugin from crashing the entire security suite, Zenith-Sentry implements strict error isolation:\n-   Each plugin is loaded within a `try-except` block.\n-   If a plugin fails to initialize or analyze, the error is logged, but the scan continues with the remaining healthy plugins.\n-   Findings from failed plugins are marked with an \"incomplete\" flag in the internal logs.\n\n---\n\n## Technical Thesis 3: Active Mitigation Protocols\n\n```\n                    ┌────────────────────┐\n                    │   Risk Detected    │\n                    └──────────┬─────────┘\n                               │\n                               ▼\n                    ┌────────────────────┐\n                    │   Risk Level ? The mitigation layer is designed to bridge the \"Time to Response\" gap.\n\n### Process Containment Strategy (SIGKILL)\n\nWhen a `CRITICAL` risk is identified by the kernel monitor (such as an unauthorized binary execution from `/tmp`), the system can immediately trigger a containment event.\n- **Why SIGKILL?**: Unlike `SIGTERM`, which can be intercepted or ignored by a malicious binary, `SIGKILL` (signal 9) is handled directly by the kernel's scheduler. The process is terminated immediately, and its memory space is reclaimed. This is the most reliable way to stop an active malware payload in its tracks.\n\n### Network Isolation Mechanics (IPTables)\n\nIn addition to process termination, Zenith-Sentry can isolate the host from a specific C2 destination.\n- **Mechanism**: The mitigation manager executes a system call to `iptables`.\n- **Logic**: It appends a `DROP` rule for the specific malicious IP detected:\n  `iptables -I OUTPUT 1 -d [MALICIOUS_IP] -j DROP`\n- **Immediate Effect**: By inserting at position `1` in the `OUTPUT` chain, we ensure that the block takes precedence over all other configured rules. This effectively \"shuts the door\" on any exfiltration attempt.\n\n### Safety Latency & Race Conditions\n\nThe mitigation engine is designed to be cautious. It verifies the existence of the process and the validity of the IP before acting. To prevent accidental system lockouts, it includes:\n- **Whitelisting**: Critical system PIDs (like PID 1) and IPs (like DNS resolvers) are protected from automated mitigation.\n- **Circuit Breakers**: If the mitigation engine triggers more than 10 times in a 60-second window, it automatically disables itself and alerts the administrator, as this likely indicates a false positive firestorm.\n\n---\n\n## Technical Thesis 4: Telemetry Collection Mechanics\n\n```\n  COLLECTORS                    SOURCES                    OUTPUT\n\n  ┌──────────────────┐ `zenith/collectors.py` performs a deep-dive into the Linux `/proc` filesystem via the `psutil` library.\n1.  **Argument Enumeration**: It extracts the full command line, not just the process name. This is critical for detecting obfuscated pipes (e.g., `curl ... | bash`).\n2.  **Context Mapping**: It correlates every process to its effective UID and GID, enabling its \"Unauthorized Root Process\" detector.\n3.  **Error Resilience**: It is designed to handle \"Racy\" processes that may exit between enumeration and inspection, using robust exception handling to prevent scan failures.\n\n### Socket & Interface Enumeration\n\nThe `NetworkCollector` monitors active connections by querying the system's network state.\n- **Capabilities**: Detects established TCP connections, listening sockets, and UDP endpoints.\n- **Noise Reduction**: Automatically filters out loopback (127.0.0.1) and internal IPC traffic to focus on outbound \"reach-back\" attempts.\n- **Attribution**: Maps varje socket back to a process ID (PID), allowing the detection engine to see *which* application is making the connection.\n\n### Persistence Artifact Deep-Scanning\n\nPersistence is the holy grail for an attacker (MITRE T1037, T1543). The `SystemCollector` recursively scans system-critical directories:\n- `/etc/systemd/system/`: To find unauthorized services.\n- `/etc/cron.d/`: To find hidden scheduled tasks.\n- `/etc/rc.local`: To identify ancient but effective persistence tricks.\n\nIt collects file metadata (size, owner, modification time, permissions), which is then analyzed for anomalies (e.g., a root-owned script in a user directory).\n\n---\n\n## Technical Thesis 5: Behavioral vs. Signature Analysis\n\nMost traditional security tools rely on **Signatures**—hashes or byte patterns of known malware. While effective against legacy threats, signature analysis fails against:\n1.  **Polymorphic Malware**: Payloads that change their hash on every execution.\n2.  **Living-off-the-Land (LotL) Attacks**: Using legitimate system tools (`curl`, `bash`, `ssh`) for malicious purposes.\n3.  **Zero-Day Exploits**: New attacks that have no existing signature.\n\nZenith-Sentry focuses on **Behavioral Analysis**. We don't care what the file \"looks like\" on disk; we care what it \"does\" when it runs. If a binary is trying to pipe a web download directly into a shell, that is inherently suspicious regardless of the binary's hash. This approach is significantly more robust against modern, state-sponsored attack chains.\n\n---\n\n## Technical Thesis 6: TUI Rendering Optimization\n\nThe Interactive TUI (`gui.py`) is built using the Python `curses` library. To keep the interface responsive during heavy scans:\n- **Asynchronous Updates**: We use non-blocking keyboard input and periodic screen refreshes.\n- **Color-Coding Latency**: Risk levels are calculated and cached per-finding to avoid redundant computation during TUI scrolling.\n- **Buffer Management**: The results window uses a pad virtual coordinate system, allowing it to display thousands of findings without memory or performance degradation.\n\n---\n\n## Technical Thesis 7: The Mitigation Manager Event Loop\n\n```\n                          ●  START\n
